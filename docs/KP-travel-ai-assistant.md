## 1. Резюме

Мы предлагаем построить под ключ **AI-ассистента в Telegram** для туристической компании партнёра — на той же архитектурной базе, которая уже работает в Wave Match (Wave Match — сервис нетворкинга, обработавший >5000 диалогов за 2026).

Система состоит из **двух связанных модулей**:

1. **AI-Консультант** — бот в Telegram, который ведёт клиента от первого касания до брони, обучаясь на тоне голоса компании и работая в режиме «оператор-в-петле» → автономно после достижения 90% точности.
2. **AI-Турподбор** — модуль, который из API партнёра подбирает 3 индивидуальных тура в бюджет клиента + 1 премиальный + релевантные **групповые туры как мини-комьюнити по интересам** и собирает всё это в красивую презентацию для клиента.

**Ключевая ценность:** партнёр получает не просто чат-бота, а **самообучающегося сейлз-менеджера 24/7**, который через 6-10 недель закрывает 90%+ диалогов без оператора, а 10% сложных кейсов передаёт человеку с полным контекстом.

---

## 2. Проблема и решение

| Текущая боль | Решение |
|---|---|
| Менеджеры повторяют одни и те же ответы по 50 раз в день | Бот обучается на этих ответах и берёт их на себя |
| Клиент уходит в ночное время, когда менеджера нет | Бот отвечает 24/7 в тоне компании |
| Подбор тура — 30-60 мин ручной работы по каталогу | Бот за 30 секунд предлагает 3+1 вариант + групповые |
| Групповые туры сложно продавать через текст | Бот подаёт их как «клуб по интересам», не «остатки» |
| Менеджер забывает что обсуждал с клиентом неделю назад | Вся история в БД, бот возвращается к контексту мгновенно |
| Нет аналитики «что спрашивают чаще всего» | Все диалоги → автоматическая еженедельная сводка |

# МОДУЛЬ 1 · AI-Консультант

## 1.1 Что делает

Telegram-бот (на основе userbot-сессии или официального BotAPI — на выбор партнёра), который:

- **Отвечает клиентам в DM 24/7** в тоне голоса компании
- **Помнит весь контекст** диалога и профиль клиента (city, бюджет, предыдущие туры, предпочтения)
- **Сначала работает с оператором-в-петле**: бот генерирует черновик → оператор одобряет реакцией ⚡/💩 → отправка / переделка
- **Учится с каждой коррекцией оператора** (Learned Scenarios)
- **Переходит в автономный режим** для каждого типа диалога, как только точность по нему >90%
- **Эскалирует к человеку** в нестандартных кейсах (политика, спорные ситуации, претензии)

## 1.2 Источники данных

| Источник | Зачем |
|---|---|
| **Tours API партнёра** | актуальные туры, цены, наличие, описания |
| **БД переписки** (PostgreSQL) | вся история диалогов, поиск похожих кейсов |
| **Tone of Voice** (документ компании) | стиль ответов, словарь, табу |
| **Learned Scenarios** (растёт автоматически) | проверенные временем шаблоны для типовых ситуаций |
| **Profile клиента** | город, бюджет, дети, аллергии, предыдущие туры, интересы |

## 1.3 Жизненный цикл обучения (стандартный, 4 фазы)

```
Phase 1: Algorithm      → пишем базовые правила (ToV, табу, эскалация)
Phase 2: Test in dialog → проверка на исторических диалогах
Phase 3: Real cases     → 2-4 недели работа с оператором, бот предлагает черновики
Phase 4: Auto-rules     → при точности >90% по сценарию — переключение в auto
```

**Метрика готовности к авто-режиму:** 9 из 10 черновиков подряд одобряются оператором без правок → сценарий помечается как `AUTO_READY`. Можно настроить порог.

## 1.4 Tone of Voice — как хранится и применяется

- **Документ ToV компании** (Word/Markdown) → парсится → структура в БД:
  - `Можно говорить:` (список разрешённых фраз/конструкций)
  - `Нельзя говорить:` (табу-слова, корпоративные клише)
  - `Примеры ответов:` (5-20 эталонных диалогов)
  - `Голос:` ты/Вы, эмодзи да/нет, формальность 1-5
- В каждый prompt к LLM подмешивается top-K релевантных правил ToV для текущей стадии диалога

## 1.5 Operator Dashboard

Веб-панель для оператора с:

- Список диалогов (открытые, ждут одобрения, в авто-режиме)
- Карточка клиента (история, профиль, теги)
- Черновики на одобрение — реакция ⚡ (отправить) / ✏️ (переписать) / 💩 (переделать с нуля)
- Метрики: точность бота по сценариям, % авто-диалогов, конверсия в бронь
- Поиск по истории «как мы отвечали на похожий вопрос»

## 1.6 Эскалация к человеку (важно!)

Бот **молчит** (не отвечает «учту, передам менеджеру») в случаях:
- Жалоба / претензия / возврат
- Запрос, которого нет в Learned Scenarios + auto-mode unreliable
- Прямая просьба клиента «дайте человека»
- Тематика вне туризма (политика, советы по жизни)

Параллельно оператору летит DM с контекстом, оператор отвечает лично → ответ автоматически становится новым тренировочным примером.

## 1.7 Технические компоненты (Модуль 1)

| Компонент | Стек |
|---|---|
| Telegram-listener | GramJS (userbot) или node-telegram-bot-api (BotAPI) |
| LLM | OpenAI GPT-4o / Claude Sonnet 4.6 (на выбор) |
| БД | PostgreSQL (Prisma ORM) |
| Очередь черновиков | Внутренняя, in-memory + persisted Approval table |
| Учёба | Cron-анализатор раз в сутки + Online (с каждого override-а оператора) |
| Веб-панель | React + Fastify backend |
| Деплой | Railway / любой VPS |

# МОДУЛЬ 2 · AI-Турподбор и презентация

## 2.1 Что делает

Когда клиент в диалоге сказал, чего хочет (или это уже в его профиле), модуль:

1. Парсит запрос → структурирует: `{бюджет, даты, страна?, тип отдыха, состав группы, особые требования}`
2. Дёргает API партнёра, фильтрует → получает кандидатов
3. **Алгоритм 3+1+G**:
   - **3 индивидуальных тура в бюджет** (разные стили: бюджетный / оптимальный / комфорт)
   - **1 премиальный** (×1.5-2 от бюджета — «если позволите себе»)
   - **G групповые туры** (1-3 шт.), которые подаются НЕ как «дешёвый вариант», а как **клуб по интересам**
4. Каждый вариант — «расшифрован»: почему именно этот, что особенного
5. Собирает всё в **красивую презентацию** (PDF / HTML / Telegram-карусель — на выбор)

## 2.2 Алгоритм «3+1+G» (детально)

### 3 индивидуальных в бюджет
- Фильтр по `price ≤ budget`, по датам, по типу
- Кластеризация по стилям (бюджет / оптимум / комфорт) — берём по 1 из каждого
- Скоринг по совпадению интересов клиента из профиля

### 1 премиум
- `price = budget × 1.5..2.0`
- Лучший по скорингу из премиум-сегмента
- Подача: «Если рассматриваете чуть дороже — обратите внимание на этот вариант, потому что…»

### G групповые туры (1-3 шт.)
- Фильтр: совпадение дат ± 7 дней, страна/тип отдыха в зоне интереса
- Скоринг по совпадению интересов с уже зарегистрированными участниками группы
- **Подача как клуб по интересам:** «В этом туре уже едут 4 человека, кому близки нетворкинг и горные походы. Возможно, найдёте свою компанию»
- Это снимает у клиента ассоциацию «групповой = эконом» и переводит в «социальная ценность»

## 2.3 Анализ запроса клиента (NLU)

Бот извлекает из свободного текста и истории:
- **Жёсткие параметры:** бюджет (диапазон), даты, страна, кол-во людей, дети
- **Мягкие параметры:** «хочу горы» / «море» / «спокойно» / «вечеринки» / «фотогенично»
- **Контекст:** «как в прошлом году в Грузии» (тянет из истории клиента)
- **Интересы:** хобби из профиля (горные походы, кулинария, йога, психология, бизнес)

Если параметров мало — бот сначала задаёт 2-3 уточняющих вопроса (не больше — не утомляет).

## 2.4 Генератор презентации

**Форматы (на выбор партнёра, можно все три):**

| Формат | Когда показывать |
|---|---|
| **Telegram-карусель** (4-5 карточек с фото + текстом + кнопками) | Быстрый показ внутри чата, для импульсных решений |
| **PDF** (А4, 4-8 страниц, фирменный стиль партнёра) | Для серьёзных решений, можно показать жене/мужу/боссу |
| **HTML лендинг** с уникальной ссылкой (`tours.partner.ru/p/abc123`) | Для расшаривания, аналитика по открытиям/кликам |

**Структура презентации (PDF/HTML):**

1. **Cover:** «Подобрали для Анны — Турция в июне до 200к₽»
2. **Tour 1-3:** карточка с фото, ценой, ключевыми точками, «почему именно вам»
3. **Premium tour:** «Если рассмотрите чуть выше бюджет — обратите внимание»
4. **Group tour:** «А ещё — есть тур, где едут люди со схожими интересами, возможно, найдёте компанию»
5. **CTA:** «Забронировать» / «Обсудить с менеджером» / «Другие варианты»

Стиль — фирменный дизайн партнёра (логотип, цвета, шрифты). Шаблон делается один раз, потом подставляются данные.

## 2.5 Технические компоненты (Модуль 2)

|---|---|

# 4. Состав работ и сроки

## Этап 0 · Discovery (1 неделя)
- Аудит существующего API партнёра, маппинг полей
- Получение ToV-документа компании, разбор
- Проектирование структуры профиля клиента
- Согласование дизайн-шаблона презентации
- **Deliverable:** ТЗ v1.0, согласовано обеими сторонами

## Этап 1 · MVP консультанта (3 недели)
- Подключение Telegram-сессии
- Базовый LLM-ответчик с ToV
- БД истории + Operator Dashboard (минимальный)
- Operator approval loop (⚡/💩)
- **Deliverable:** бот отвечает оператору на 100% диалогов, оператор одобряет

## Этап 2 · Turподбор + презентации (3 недели)
- Адаптер к Tours API партнёра
- NLU извлечения параметров
- Алгоритм 3+1+G
- Генератор презентации (один формат на выбор — PDF / TG-карусель)
- **Deliverable:** бот по запросу подбирает туры и собирает презентацию

## Этап 3 · Обучение и автономия (2-4 недели)
- Cron-анализатор learned scenarios
- Auto-режим переключения по сценариям
- Эскалация к оператору
- Дашборд метрик
- **Deliverable:** 60-90% диалогов в автономном режиме (зависит от качества ToV-данных)

## Этап 4 · Расширения (опционально, по подписке)
- Дополнительные форматы презентаций
- Интеграция с CRM партнёра (AmoCRM, Bitrix24)
- Голосовые ответы клиенту
- Брони онлайн через @wallet (как в Wave Match)

**Итого:** MVP + Турподбор за **6-7 недель**. Автономия + дашборд — ещё **2-4 недели**.
